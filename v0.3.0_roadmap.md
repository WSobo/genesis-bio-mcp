## Milestone 1 — Ensembl REST + VEP integration

### Rationale
`get_variant_effects` currently resolves protein changes through gnomAD's variant catalog and falls back to MyVariant's query endpoint when gnomAD has no hit. This works for missense variants present in population data but has three real weaknesses:

1. No transcript isoform handling — a mutation in a minor isoform won't resolve cleanly
2. No genomic coordinate anchoring — everything is HGVS-string based
3. No consequence annotation beyond missense — splice, UTR, regulatory overlap is invisible

Ensembl REST solves all three with one API.

### Files to add
- `src/genesis_bio_mcp/clients/ensembl.py` — new client
- `src/genesis_bio_mcp/models.py` — add `EnsemblGene`, `VEPConsequence`, `TranscriptInfo` models
- `tests/unit/test_ensembl.py` — unit tests with recorded fixtures
- `tests/integration/test_ensembl_live.py` — 3–5 live API checks

### Files to modify
- `src/genesis_bio_mcp/clients/variant_effects.py` — inject EnsemblClient; add VEP consequence step
- `src/genesis_bio_mcp/tools/gene_resolver.py` — add Ensembl as fallback resolution path
- `src/genesis_bio_mcp/server.py` — register client in lifespan, add new tool `get_variant_consequences`

### API endpoints to wrap
- `GET /lookup/symbol/homo_sapiens/{symbol}` — gene → Ensembl ID + coordinates
- `GET /overlap/region/human/{region}` — features overlapping a genomic window
- `GET /vep/human/hgvs/{hgvs}` — variant consequence prediction
- `GET /vep/human/region/{region}/{allele}` — VEP by coordinate (when HGVS unavailable)
- `GET /sequence/id/{id}?type=cdna` — transcript sequence for splice-site analysis

### New tool: `get_variant_consequences`
Distinct from `get_variant_effects` (which is about pathogenicity) — this one is about *what the variant hits*: splice donor/acceptor, UTR, regulatory region, transcript impact. Returns VEP consequences with SIFT/PolyPhen from Ensembl's own annotation (complementary to the dbNSFP values MyVariant returns).

Input: `{gene_symbol, mutation}` or `{hgvs_genomic}` or `{chrom, pos, ref, alt}`

Output: `VEPConsequenceReport` with most severe consequence, all transcript effects, overlap with regulatory features.

### Integration points
In `VariantEffectsClient.get_effects()`, add a third parallel task alongside MyVariant and MaveDB:
```python
vep_task = self._ensembl.get_vep_consequences(gene_symbol, mutation)
annotation, dms_scores, vep = await asyncio.gather(
    annotation_task, dms_task, vep_task
)
```
Add `vep_consequences` field to `VariantEffects` model.

### Gotchas
- Ensembl's rate limit is 15 req/s burst, 55k/hr sustained — semaphore at 5 is safe
- VEP responses are large (every transcript × every consequence); filter to canonical transcript by default, add `include_all_transcripts: bool = False` param
- HGVS format must be strict — Ensembl rejects `p.Arg175His` without a transcript accession prefix, so keep the existing `parse_protein_change` as the normalization step and build the Ensembl query from canonical Ensembl transcript ID resolved in the gene lookup
- The 500 errors Ensembl occasionally returns on VEP are transient — add retry with exponential backoff

### Effort estimate
- Client: 2 days
- Variant effects integration: 1 day
- New tool + tests: 1.5 days
- **Total: ~5 days**

## Milestone 2 — GTEx + Human Protein Atlas

### Rationale
You have no quantitative expression data. Target ID workflows consistently need:
- "Is this gene expressed in the tissue where the disease manifests?"
- "Is expression restricted enough to give a therapeutic window?"
- "What other tissues express it — where might I see toxicity?"

GTEx covers bulk RNA across ~54 tissues from ~1000 donors. HPA adds protein-level immunohistochemistry, subcellular localization imaging, and pathology staining across tumors. They pair naturally.

### Files to add
- `src/genesis_bio_mcp/clients/gtex.py`
- `src/genesis_bio_mcp/clients/hpa.py`
- Models: `GTExExpression`, `TissueExpressionProfile`, `HPAExpression`, `HPAPathologyData`
- Tests for both

### GTEx endpoints
- `GET /api/v2/expression/medianGeneExpression?gencodeId={id}` — median TPM per tissue
- `GET /api/v2/reference/gene/{gene_id}` — gene metadata

GTEx uses GENCODE IDs, not HGNC symbols — add a resolution step using Ensembl client from M1 (or cache a static HGNC→GENCODE map if M1 slips).

### HPA endpoints
HPA's API is less RESTful; it's XML dumps keyed on Ensembl ID:
- `GET /api/search_download.php?search={symbol}&format=json&columns=g,gs,eg,rnatsm,rnats,scml,scl,prognostic_cancer`

Returns tissue specificity category, enhanced tissues, subcellular locations, and prognostic data for cancer outcomes. Parse once, cache per-gene.

### New tools
- `get_tissue_expression` — GTEx median TPM per tissue + HPA tissue specificity category
- `get_protein_atlas` — HPA subcellular IHC, pathology staining, prognostic cancer survival data

### Integration into `prioritize_target`
Add an expression axis to the scoring breakdown. Proposed:
- **Tissue specificity score (max 1.0):** bonus when expression is enriched in the indication's tissue. Uses HPA's `tissue_specificity_category` field: "Tissue enriched" = 1.0, "Group enriched" = 0.7, "Tissue enhanced" = 0.5, "Low tissue specificity" = 0.2, "Not detected" = 0.0
- Requires indication → tissue mapping (can be a hardcoded dict for v0.3 covering the top 20 therapeutic areas, or resolved via Ensembl EFO → UBERON mapping)

Update `ScoreBreakdown` model to include `expression` field. Adjust total max from 10.0 to 11.0 or rebalance the existing axes (I'd rebalance — keeping 10.0 as the cap is cleaner for users).

### Gotchas
- GTEx uses v10 GENCODE IDs; genes without a GENCODE mapping return 404
- HPA data is slow to fetch (~2s per gene uncached) — 7-day disk cache using the same pattern as SAbDab
- HPA XML has ~40 fields; only pull what's actually used in markdown output or score computation
- Expression "enrichment" in tumor tissue is not the same as "expression in disease tissue" — be careful about how the score interprets HPA pathology data vs. GTEx normal tissue

### Effort estimate
- GTEx client: 1.5 days
- HPA client: 2 days (harder API)
- Tools + integration: 1.5 days
- Scoring rebalance: 1 day
- **Total: ~6 days**

## Milestone 3 — OpenFDA FAERS + drug history upgrade

### Rationale
`get_drug_history` currently answers "what drugs exist for this target?" but not "how safe have they been?" Adding FAERS adverse event reports closes this. It also future-proofs the tool for safety signal work — if you join Genesis or Iambic and someone asks "what's the AE profile for class X kinase inhibitors?", this gives a clean answer.

### Files to add
- `src/genesis_bio_mcp/clients/openfda.py`
- Models: `AdverseEventReport`, `DrugSafetySignal`

### OpenFDA endpoints
- `GET /drug/event.json?search=patient.drug.medicinalproduct:{drug_name}&count=patient.reaction.reactionmeddrapt.exact` — reaction frequency for a named drug
- `GET /drug/label.json?search=openfda.generic_name:{drug_name}` — FDA label including boxed warnings
- `GET /drug/enforcement.json?search=product_description:{drug_name}` — recalls

No API key required for light use (240 req/min, 1000 req/day without a key; much higher with one — worth requesting).

### Integration into `get_drug_history`
Add a safety signal step per known approved drug:
```python
safety_signals = await asyncio.gather(*[
    self._openfda.get_safety_signals(drug.drug_name)
    for drug in drugs if drug.approved
])
```
Cap at top 5 drugs by phase to bound latency. Add `boxed_warnings: list[str]` and `top_adverse_events: list[tuple[str, int]]` fields to `DrugInteraction`.

### Gotchas
- FAERS reports are noisy — a single unusual cluster can spike one MedDRA term. Use proportional reporting ratio (PRR) if you want to surface signals, or just report raw counts with a caveat
- Brand name vs. generic name matching is messy; try generic first, fall back to brand
- Drug recalls are rare but when they matter they really matter — surface them prominently
- Don't present FAERS data as clinical advice; the output model should include a disclaimer string

### Effort estimate
- Client: 2 days
- Tool integration: 1 day
- Tests: 0.5 days
- **Total: ~3.5 days**

## Milestone 4 — ChEMBL assay context upgrade

### Rationale
Not a new API — a depth fix on the existing `ChEMBLClient`. Right now `get_chembl_compounds` returns pChEMBL values without assay context, so a pChEMBL 9 biochemical binding assay and a pChEMBL 9 cell-based functional assay are presented identically. For anyone making real decisions from the data, this is a significant limitation.

### Files to modify
- `src/genesis_bio_mcp/clients/chembl.py` — expand `_fetch_activities` field selection
- `src/genesis_bio_mcp/models.py` — add fields to `ChEMBLActivity`

### Fields to add
Pull these from the ChEMBL activity endpoint (already in the response, just not parsed):
- `assay_type` (B = binding, F = functional, A = ADME, T = toxicity, P = physchem)
- `assay_organism` (human, rat, mouse — selectivity matters)
- `assay_cell_type` (cell line name for functional assays)
- `bao_format` (BAO ontology term — single-protein format, cell-based, tissue-based)
- `confidence_score` (0–9; 9 = direct single-protein target)

### Output changes
Update `to_markdown` to show assay type and organism alongside potency. Consider grouping table rows by assay type so biochemical and cell-based hits are visually distinct.

### Scoring impact
Downgrade chem_matter contribution slightly when all potency data comes from non-human assays or low-confidence targets. Minor change, but closes an obvious credibility gap.

### Effort estimate
- **Total: ~2 days**

## Architectural notes

### Client pattern consistency
All new clients follow the existing pattern in your codebase:
- Accept `httpx.AsyncClient` in constructor
- Session-scoped cache dict keyed by normalized input
- Module-level semaphore for rate limiting
- All methods return `None` on failure, never raise
- Disk cache via `settings.{name}_cache_path` when payload is heavy

### Testing strategy
Your existing split (unit with fixtures / integration live) works well. For v0.3.0 each new client needs:
- Recorded JSON fixtures from at least 3 representative genes (BRAF, TP53, CFTR cover oncology + tumor suppressor + disease gene patterns)
- One live smoke test tagged `@pytest.mark.integration` that actually hits the API
- Schema validation tests verifying Pydantic models parse real responses

### Breaking changes to flag
- Scoring scale change (if done) affects any automation calling `prioritize_target` and parsing scores
- `ScoreBreakdown` field additions are non-breaking (additive) but tier thresholds may shift
- `ChEMBLActivity` field additions are non-breaking

### Deferred to v0.4.0
Based on the earlier gap analysis:
- ProteomicsDB / CPTAC mass-spec protein abundance
- OMIM / ClinGen curated Mendelian disease associations
- Patent landscape (USPTO/Lens.org)
- AlphaFold Multimer / complex prediction
- ESM Metagenomic Atlas

These are all legitimate additions but each one is a self-contained client and the v0.3 scope is already generous.

## Open questions before starting

1. **Scoring scale** — keep 10.0 cap and rebalance, or move to 12.0? I'd argue rebalance; rebalance preserves user mental models
2. **Indication → tissue mapping** — hardcoded v0.3 dict or EFO→UBERON resolution via Ensembl? Hardcoded is faster to ship; ontology-backed is more correct. Suggest starting hardcoded and migrating in v0.3.1
3. **OpenFDA rate limit key** — request one now; free, fast approval, and 240 req/min without it will bite during development
4. **VEP canonical-only default** — strong yes. Users who need all transcripts can pass the flag; most queries need one answer
5. **HPA cache storage** — HPA data is fairly stable quarterly; 7-day disk cache matches SAbDab and is a reasonable default
